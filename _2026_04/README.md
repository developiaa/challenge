# PLCC 카드 발급 비동기 파이프라인 설계서

본 문서는 Spring WebFlux, R2DBC, Kotlin Coroutines 환경에서 대용량 트래픽과 외부 API 장애를 방어하기 위해 설계된 PLCC 카드 발급 파이프라인의 핵심 기술 구조를 설명합니다.

---

## 1. 개요 및 아키텍처 철학

이 시스템은 수만 명의 동시 접속자가 몰려도 서버의 스레드와 DB 커넥션 풀이 고갈되지 않도록 **100% 논블로킹(Non-blocking)**으로 설계되었습니다. 기존 `@Transactional`이 가진 한계를 극복하기 위해 **트랜잭션 경계를 명확히 분리**하였으며, 상황에 따라 순차 처리(Sequential)와 병렬 처리(Parallel) 두 가지 전략을 제공합니다.

---

## 2. 단일 외부 연동 파이프라인 (`issuePlccCard`)

하나의 외부 카드사 API를 호출하고 결과를 반영하는 가장 표준적인 비동기 파이프라인입니다.

### 📍 흐름도 (Execution Flow)
1. **[진입]** 분산 락(Redis)을 통한 중복 요청 검증 (따닥 방지).
2. **[Tx 1]** DB에 발급 이력을 `PENDING` 상태로 저장하고 **트랜잭션 종료 (커넥션 반환)**.
3. **[I/O]** 외부 카드사 API 호출 (스레드 블로킹 없이 `suspend` 상태로 대기).
4. **[Tx 2]** 응답 결과(`0000`)에 따라 새로운 트랜잭션을 열어 DB 상태를 `SUCCESS` 또는 `FAIL`로 업데이트.

### 💡 핵심 기술 포인트
* **트랜잭션 경계 분리 (Transaction Boundary Separation):** 외부 API 호출 시점에 DB 커넥션을 물고 있지 않습니다. 외부 카드사 서버가 지연(장애)을 일으켜 응답이 3초~5초 지연되더라도, 내부 시스템의 DB 커넥션 풀은 안전하게 보호됩니다.
* **스레드 스위칭 (Thread Switching):** I/O 대기 시간 동안 이벤트 루프 스레드는 다른 요청을 처리합니다. 로그를 확인해 보면 `[Step 3]`과 `[Step 4]` 사이에서 코루틴을 실행하는 스레드(`reactor-http-nio-X`)가 동적으로 변경되는 것을 확인할 수 있습니다.
* **보상 트랜잭션 (Compensating Transaction):** 네트워크 타임아웃이나 예외 발생 시 `catch` 블록에서 즉시 상태를 `FAIL`로 업데이트하여 데이터 정합성을 유지합니다.

---

## 3. 다중 외부 연동 병렬 파이프라인 (`issuePlccCard2`)

여러 외부 API(예: 카드사 인증 API + 발급 API)를 **동시에** 검증해야 할 때 사용하는 고성능 파이프라인입니다.

### 📍 흐름도 (Execution Flow)
1. **[Tx 1]** DB에 발급 이력을 `PENDING` 상태로 저장.
2. **[I/O]** `async` 블록을 활용하여 두 개의 외부 API를 **병렬(Parallel)로 동시 호출**.
3. **[대기]** `awaitAll()`을 통해 두 API의 응답이 모두 도착할 때까지 논블로킹 대기.
4. **[Tx 2]** 모든 검증이 통과되면 `SUCCESS`로 DB 업데이트.

### 💡 핵심 기술 포인트
* **구조화된 동시성 (Structured Concurrency):** `coroutineScope` 내부에서 실행되는 코루틴들은 부모-자식 관계로 묶입니다. 두 API 호출 중 하나라도 예외를 던지면, 부모 스코프가 즉시 **나머지 진행 중인 API 호출을 취소(Cancel)**합니다. 불필요한 네트워크 대기와 리소스 낭비를 원천 차단하는 'Fast-fail' 메커니즘입니다.
* **소요 시간 최적화:** 각각 5초가 걸리는 API 2개를 호출하더라도, 직렬로 호출하면 10초가 걸리지만 `async`를 활용하면 가장 늦게 끝나는 API의 시간(5초)만큼만 소요됩니다. 대용량 트래픽에서 응답 지연(Latency)을 획기적으로 줄일 수 있습니다.

---

## 4. 파이프라인 비교 요약

| 구분 | `issuePlccCard` (순차) | `issuePlccCard2` (병렬) |
| :--- | :--- | :--- |
| **주요 용도** | 단일 외부 API 연동 | 2개 이상의 독립적인 외부 API 동시 연동 |
| **호출 방식** | 동기적 흐름의 논블로킹 (`suspend`) | 병렬 논블로킹 (`async` + `awaitAll`) |
| **예외 전파** | 단일 예외 발생 시 즉시 `catch` | 하나의 `async` 실패 시 다른 `async`도 즉시 취소 (Fast-fail) |
| **소요 시간** | 외부 API 소요 시간과 동일 | 호출된 API 중 가장 오래 걸리는 시간과 동일 |
| **트랜잭션 관리** | 분리 완료 (DB 풀 고갈 방지) | 분리 완료 (DB 풀 고갈 방지) |
